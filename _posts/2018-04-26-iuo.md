올해 초 Swift 컴파일러에 암시적 추출 옵셔널(IUO)의 새로운 구현이 반영되었으며, 최근 Swift [스냅샷](/download/#snapshots)에서 사용해 볼 수 있습니다.
이로써 [SE-0054 - ImplicitlyUnwrappedOptional 타입 폐지](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0054-abolish-iuo.md)의 구현이 완료되었습니다.
이는 타입 검사의 일부 불일치를 제거하고, 이러한 값을 어떻게 다룰지에 대한 규칙을 일관되고 이해하기 쉽게 명확히 한 중요한 언어 변경입니다. 자세한 내용은 해당 제안의 [동기 섹션](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0054-abolish-iuo.md#motivation)을 참고하세요.

가장 눈에 띄는 변경 사항은 진단 메시지에서 기반 타입이 `T`인 암시적 추출 옵셔널로 선언된 값을 참조할 때 `T!` 대신 `T?`를 출력한다는 것입니다.
코드를 수정해야 성공적으로 컴파일할 수 있는 소스 호환성 문제가 발생할 수도 있습니다.

## 암시적 추출은 선언의 일부

*암시적 추출 옵셔널*은 표현식이 컴파일되기 위해 필요할 때 자동으로 언래핑되는 옵셔널입니다. 암시적으로 추출되는 옵셔널을 선언하려면 타입 이름 뒤에 `?` 대신 `!`를 붙입니다.

많은 사람들이 암시적 추출 옵셔널에 대해 갖고 있는 멘탈 모델은 일반 옵셔널과 구별되는 별도의 타입이라는 것입니다. Swift 3에서는 실제로 그렇게 동작했습니다: `var a: Int?` 같은 선언은 `a`의 타입이 `Optional<Int>`이 되고, `var b: String!` 같은 선언은 `b`의 타입이 `ImplicitlyUnwrappedOptional<String>`이 되었습니다.

IUO에 대한 새로운 멘탈 모델에서는 `!`를 `?`의 동의어로 보되, 선언된 값이 암시적으로 추출될 수 있음을 컴파일러에 알리는 플래그를 선언에 추가하는 것으로 생각하면 됩니다.

즉, `String!`을 "이 값은 `Optional<String>` 타입이며, 필요할 때 암시적으로 추출될 수 있다는 정보도 함께 가지고 있다"고 읽을 수 있습니다.

이 멘탈 모델은 새로운 구현과 일치합니다. `T!`가 있는 모든 곳에서 컴파일러는 이제 이를 `T?` 타입으로 취급하고, 필요한 곳에서 값을 암시적으로 추출할 수 있음을 타입 검사기에 알리는 플래그를 선언의 내부 표현에 추가합니다.

이 변경의 가장 눈에 띄는 결과는 `T!`로 선언된 값에 대해 진단 메시지에서 `T!` 대신 `T?`로 표시된다는 것입니다. 진단에서 `T!` 대신 `T?`를 보는 것이 처음에는 적응이 필요하지만, 이 새로운 멘탈 모델을 받아들이면 도움이 될 것입니다.

## 소스 호환성

대부분의 프로젝트는 호환성 문제 없이 빌드됩니다. 하지만 이러한 구현 변경으로 인해 SE-0054와는 일치하지만 이전 컴파일러 릴리스와는 일치하지 않는 동작 변경이 발생할 수 있습니다.

### T!로의 강제 변환

`as T!` 형태의 강제 변환은 SE-0054에 의해 허용되지 않습니다.

Swift 4.1에서는 이러한 강제 변환에 대해 지원 중단 경고가 발생합니다. 대부분의 경우 `as T!`를 `as T?`로 바꾸거나 강제 변환을 제거하면 컴파일에 성공합니다.

기존 코드가 이 두 가지 변경 중 하나로 컴파일에 실패하는 경우가 충분히 있어서, 새 구현에서는 이에 대한 특별 처리가 있습니다. 구체적으로, `x as T!`를 작성하면 컴파일러는 먼저 이를 `x as T?`로 타입 검사를 시도합니다. 그것이 실패한 경우에만 컴파일러는 옵셔널을 강제 추출하는 `(x as T?)!`로 타입 검사를 시도합니다.

하지만 이 형태의 강제 변환은 여전히 지원 중단으로 간주되며, 이 특별 처리는 향후 Swift 버전에서 제거될 수 있습니다.

### 선언이 아닌 타입에 ! 사용

`T!`로의 강제 변환은 더 일반적인 문제의 특수한 경우입니다: 타입의 일부로 `!`를 사용하는 것.

타입의 일부로 `!`를 사용할 수 있는 곳은 세 군데입니다:

1. 프로퍼티 선언
2. 함수 선언의 *매개변수*
3. 함수 선언의 *반환 값*

다른 위치에서는 `!`가 오류로 표시되어야 하며, Swift 4.1 이전 릴리스에서 그렇게 하려고 했지만 일부 경우를 놓쳤습니다:

~~~swift
let fn: (Int!) -> Int! = ...   // error: not a function declaration!
~~~

Swift 4.1에서는 이러한 시나리오에서 지원 중단 경고를 내보내지만 암시적 추출 동작은 계속 유지합니다. 최근 스냅샷의 새 구현에서는 `!`를 `?`처럼 취급하고 무슨 일이 일어나고 있는지, 그리고 이러한 위치에서 `!` 사용이 지원 중단되었음을 알리는 진단을 내보냅니다.

### 암시적 추출 옵셔널로 선언된 값에서 map 호출

이전에는 이런 코드가:

~~~swift
class C {}
let values: [Any]! = [C()]
let transformed = values.map { $0 as! C }
~~~

`values`를 강제 추출한 다음 배열에서 [`map(_:)`](https://developer.apple.com/documentation/swift/array/2908681-map)를 호출하는 결과가 되었습니다. `ImplicitlyUnwrappedOptional`의 익스텐션에 멤버 `map(_:)`를 정의한 경우에도 마찬가지였는데, `ImplicitlyUnwrappedOptional`에 대한 멤버 조회가 예상대로 동작하지 않았기 때문입니다.

새 구현에서는 `!`가 `?`의 동의어이므로, 컴파일러가 여기서 `Optional<T>`의 [`map(_:)`](https://developer.apple.com/documentation/swift/optional/1539476-map)를 호출하려고 시도합니다:

~~~swift
let transformed = values.map { $0 as! C } // calls Optional.map; $0 has type [Any]
~~~

그리고 다음을 생성합니다: `warning: cast from '[Any]' to unrelated type 'C' always fails`

기술적으로 타입 검사를 통과하므로, `values`를 강제 추출하려고 시도하지 않습니다.

옵셔널 체이닝을 사용하여 옵셔널 배열을 생성하는 방법으로 해결할 수 있습니다:

~~~swift
let transformed = values?.map { $0 as! C } // transformed has type Optional<[C]>
~~~

또는 `values`를 강제 추출하여 배열을 생성하는 방법도 있습니다:

~~~swift
let transformed = values!.map { $0 as! C } // transformed has type [C]
~~~

많은 경우에는 동작 변경이 없습니다:

~~~swift
let values: [Int]! = [1]
let transformed = values.map { $0 + 1 }
~~~

`Optional`에서 `map(_:)`를 호출하면 표현식을 성공적으로 타입 검사할 수 없으므로, 이전과 동일하게 동작합니다. 대신 `values`를 강제 추출하고 결과 배열에서 `map(_:)`를 호출하게 됩니다.

### 타입이 아닌 것은 타입으로 추론할 수 없다

암시적 추출 옵셔널은 더 이상 옵셔널과 구별되는 타입이 아니므로, 타입이나 타입의 일부로 추론될 수 없습니다.

아래 예제에서, 대입의 오른쪽이 암시적 추출로 선언된 값을 포함하더라도, 왼쪽에 대해 추론된 타입은 값(또는 반환 값)이 옵셔널임만을 나타냅니다.

~~~swift
var x: Int!
let y = x   // y has type Int?

func forcedResult() -> Int! { ... }
let getValue = forcedResult    // getValue has type () -> Int?

func id<T>(_ value: T) -> T { return value }
let z = id(x)   // z has type Int?

func apply<T>(_ fn: () -> T) -> T { return fn() }
let w: Int = apply(forcedResult)    // fails, because apply() returns unforced Int?
~~~

이 동작 변경을 알 수 있는 구체적인 사례로는 `AnyObject` 조회, `try?`, `switch`가 있습니다.

#### AnyObject 조회

`AnyObject` 조회의 결과는 암시적으로 추출되는 옵셔널로 취급됩니다. 그 자체가 암시적 추출로 선언된 프로퍼티를 조회하면, 표현식은 이제 두 레벨의 암시적 추출을 가집니다(`property`는 `UILabel!`로 선언됨):

~~~swift
func getLabel(object: AnyObject) -> UILabel {
  return object.property // forces both optionals, resulting in a UILabel
}
~~~

`if let`과 `guard let`은 한 레벨의 옵셔널만 언래핑합니다.

다음 예제에서, 이전 Swift 버전은 `if let`에서 한 레벨의 옵셔널을 언래핑한 후 `label`의 타입을 `UILabel!`로 추론했습니다. 스냅샷 빌드에서 Swift는 이를 `UILabel?`로 추론합니다:

~~~swift
// label is inferred to be UILabel?
if let label = object.property {
   // Error due to passing a UILabel? where a UILabel is expected
  functionTakingLabel(label)
}
~~~

명시적 타입을 사용하여 이를 해결할 수 있습니다:

~~~swift
// Implicitly unwrap object.property due to explicit type.
if let label: UILabel = object.property {
  functionTakingLabel(label) // okay
}
~~~


#### try?

마찬가지로, `try?`는 한 레벨의 옵셔널을 추가하므로, 암시적 추출 값을 반환하는 함수와 `try?`를 결합할 때 두 번째 레벨의 옵셔널을 명시적으로 언래핑하도록 코드를 수정해야 할 수 있습니다.

~~~swift
func test() throws -> Int! { ... }

if let x = try? test() {
  let y: Int = x    // error: x is an Int?
}

if let x: Int = try? test() { // explicitly typed as Int
  let y: Int = x    // okay, x is an Int
}

if let x = try? test(), let y = x { // okay, x is Int?, y is Int
 ...
}
~~~

#### switch

Swift 4.1은 `output`을 암시적 추출로 취급했기 때문에 다음 코드를 허용했습니다:

~~~swift
func switchExample(input: String!) -> String {
  switch input {
  case "okay":
    return "fine"
  case let output:
    return output  // implicitly unwrap the optional, producing a String
  }
}
~~~

이렇게 작성했다면 성공적으로 컴파일되지 않았을 것입니다:

~~~swift
func switchExample(input: String!) -> String {
  let output = input  // output is inferred to be String?
  switch input {
  case "okay":
    return "fine"
  default:
    return output  // error: value of optional type 'String?' not unwrapped;
                   // did you mean to use '!' or '?'?
  }
}
~~~

새 구현에서는 첫 번째 예제의 `output` 타입을 암시적으로 추출되지 않는 `String?`로 추론합니다.

다시 컴파일되게 하는 한 가지 방법은 값을 강제 추출하는 것입니다:

~~~swift
  case let output:
    return output!
~~~

또 다른 수정 방법은 non-nil과 nil에 대해 명시적으로 패턴 매칭하는 것입니다:

~~~swift
func switchExample(input: String!) -> String {
  switch input {
  case "okay":
    return "fine"
  case let output?: // non-nil case
    return output   // okay; output is a String
  case nil:
    return "<empty>"
  }
}
~~~

### 옵셔널 vs 암시적 추출 옵셔널 In-Out 매개변수 오버로딩

Swift 4.1에서는 in-out 매개변수가 일반 옵셔널인 것과 암시적 추출 옵셔널인 것의 차이만으로 함수를 오버로드하려는 경우에 대해 지원 중단 경고가 도입되었습니다.

~~~swift
  func someKindOfOptional(_: inout Int?) { ... }

  // Warning in Swift 4.1.  Error in new implementation.
  func someKindOfOptional(_: inout Int!) { ... }
~~~

Swift 4.1에서는 암시적 추출로 선언된 값을 일반 옵셔널을 기대하는 함수에 in-out 매개변수로 전달하는 기능과 그 반대도 추가되었습니다. 이를 통해 (구현이 동일하다는 가정 하에) 위의 두 번째 오버로드를 삭제할 수 있게 되었습니다:

~~~swift
  func someKindOfOptional(_: inout Int?) { ... }

  var i: Int! = 1
  someKindOfOptional(&i)   // okay! i has type Optional<Int>
~~~

암시적 추출 옵셔널의 새 구현에서는 `Int!`의 타입이 `Int?`의 동의어이므로, 옵셔널리티에 의한 오버로딩은 더 이상 의미가 없습니다. 결과적으로 위와 같은 오버로드는 이제 오류가 발생하며, 두 번째 오버로드(`Int!`로 선언된)를 제거해야 합니다.

### ImplicitlyUnwrappedOptional의 익스텐션

`ImplicitlyUnwrappedOptional<T>`는 이제 `Optional<T>`의 사용 불가 타입 별칭이며, 이 타입에 익스텐션을 만들려는 코드는 컴파일되지 않습니다:

~~~swift
// 1:11: error: 'ImplicitlyUnwrappedOptional' has been renamed to 'Optional'
extension ImplicitlyUnwrappedOptional {
~~~

### Nil 브리징

`nil` 값을 브리징할 때 런타임 실패가 발생하는 대신, `nil`은 `NSNull`로 브리징됩니다.

~~~swift
import Foundation

class C: NSObject {}

let iuoElement: C! = nil
let array: [Any] = [iuoElement as Any]
let ns = array as NSArray
let element = ns[0] // Swift 4.1: Fatal error: Attempt to bridge
                    // an implicitly unwrapped optional containing nil

if let value = element as? NSNull, value == NSNull() {
  print("pass")     // We reach this statement with the new implementation
} else {
  print("fail")
}
~~~

## 결론

암시적 추출 옵셔널은 더 이상 `Optional<T>`와 구별되는 타입이 아니도록 재구현되었습니다. 결과적으로 타입 검사가 더 일관되고 컴파일러의 특수 케이스가 줄었습니다. 이러한 특수 케이스를 제거함으로써 이러한 선언을 처리할 때 버그가 줄어들 것입니다.

임포트된 Objective-C API와 상호작용할 때 암시적 추출을 접하게 될 것입니다. `@IBOutlet` 프로퍼티를 선언하거나, 값이 완전히 초기화될 때까지 접근하지 않을 것을 *확신*하는 곳에서 가끔 암시적 추출이 편리할 수 있습니다. 하지만 보통은 암시적 추출을 피하고 `if let`과 `guard let`을 통한 명시적 언래핑을 사용하는 것이 좋습니다. 안전하다고 확신할 때는 `!`를 통한 명시적 강제 추출을 사용하세요.
