Swift 4.1에서 컴파일러가 코드 크기를 줄이기 위한 전용 최적화를 활성화하는 새로운 최적화 모드를 지원합니다.

Swift 컴파일러에는 강력한 최적화 기능이 내장되어 있습니다. `-O` 옵션으로 컴파일하면 컴파일러가 최대 성능으로 실행되도록 코드를 변환하려고 합니다. 하지만 이러한 런타임 성능 향상은 때때로 코드 크기 증가라는 트레이드오프를 수반할 수 있습니다.
새로운 `-Osize` 최적화 모드를 사용하면 최대 속도 대신 최소 코드 크기를 목표로 컴파일할 수 있습니다.

커맨드 라인에서 크기 최적화 모드를 활성화하려면 `-O` 대신 `-Osize`를 사용하면 됩니다. Xcode 9.3에는 새로운 Swift 컴파일러 코드 생성 빌드 설정이 추가되었습니다:

![Xcode 최적화 모드 설정](/assets/images/osize-blog/Xcode-Osize-setting.png)

또한 컴파일 모드(단일 파일 또는 전체 모듈)를 최적화 모드와 독립적으로 선택할 수 있게 되었습니다:

![Xcode 컴파일 모드 설정](/assets/images/osize-blog/Xcode-mode-setting.png)

`-Osize` 모드는 전체 모듈 컴파일과 단일 파일 컴파일 모두에서 동작하며, 전체 모듈 모드에서 최적의 결과를 얻을 수 있습니다.

일부 프로젝트에서 `-Osize`를 사용했을 때 코드 크기가 5%에서 최대 30%까지 줄어드는 것을 확인했습니다.

그렇다면 성능은 어떨까요? 이는 전적으로 프로젝트에 따라 다릅니다. 대부분의 애플리케이션에서 `-Osize`로 인한 성능 저하는 무시할 수 있는 수준, 즉 5% 미만일 것입니다. 하지만 성능에 민감한 코드에서는 `-O`가 여전히 더 나은 선택일 수 있습니다.

### 코드 최적화에 대한 영향

컴파일러가 `-Osize`에서 무엇을 다르게 하는지 자세히 살펴보겠습니다.
`-Osize`에서도 `-O`와 마찬가지로 컴파일러가 코드를 최적화합니다.
하지만 `-O`와 달리 컴파일러는 코드 중복을 피하려고 합니다. 예를 들어, 함수를 인라이닝할 때 컴파일러는 함수를 인라이닝할지 결정하는 데 더 낮은 크기 제한을 적용합니다.

함수 인라이닝을 완전히 비활성화하는 것은 좋지 않은 방법입니다. 작은 함수를 인라이닝하면 오히려 코드 크기가 개선되는 경우가 많기 때문입니다. 예를 들어 다음과 같은 단순한 getter 함수를 생각해 보세요:

~~~swift
struct X {
    var x: Int { return 27 }
}
~~~

이 getter를 호출하는 오버헤드가 함수를 인라이닝하는 것보다 훨씬 큽니다. 이것은 극단적인 예시이지만, 실제로 일정 크기까지는 인라이닝이 여전히 코드 크기를 개선하는 효과가 있습니다.
또한 함수 인라이닝은 다른 최적화를 유발할 수 있으며, 이것이 결과적으로 코드 크기를 줄일 수 있습니다. 예를 들어 아래 코드에서 getter `a.x`를 인라이닝하면 `a.x`가 27으로 평가된다는 것을 알 수 있으므로 전체 `if` 분기를 최적화하여 제거할 수 있습니다:

~~~swift
func foo(a: X) {
    if a.x != 27 {
        // a.x의 getter가 인라이닝되면 최적화로 제거 가능
    }
}
~~~

인라이닝 외에도 컴파일러는 `-Osize`에서 다른 코드 크기 관련 최적화를 수행합니다. 예를 들어, 제네릭 타입 처리나 Objective-C 브리징을 위한 일부 코드 패턴은 헬퍼 함수로 추출되어 인라인으로 생성되지 않습니다.

### 결론

새로운 `-Osize` 최적화 모드는 극단적으로 성능에 민감하지 않은 프로그램의 코드 크기를 줄이는 좋은 방법입니다.

`-Osize`를 사용해 보시고 피드백을 주시길 바랍니다. 포럼에서 [osize](https://forums.swift.org/tags/osize) 태그를 사용하여 경험을 공유해 주세요.
