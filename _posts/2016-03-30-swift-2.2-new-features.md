Swift 2.2는 새로운 구문, 새로운 기능, 그리고 일부 기능의 지원 중단(deprecation)을 포함합니다. 올해 후반 [더 큰 변경 사항](/blog/swift-api-transformation/)을 가져올 Swift 3 이전의 중간 릴리스이며, Swift 2.2의 변경 사항은 누락된 기능을 추가하고, 기존 기능을 다듬고, 언어에서 더 이상 필요하지 않은 것을 제거하여 핵심 언어와 표준 라이브러리를 점진적으로 안정화하는 Swift 3의 넓은 목표에 부합합니다. Swift 2.2의 모든 변경은 커뮤니티 주도의 [Swift Evolution 프로세스](/contributing/#participating-in-the-swift-evolution-process)를 거쳤습니다 &mdash; Swift가 몇 달 전 오픈 소스화된 이후 30개 이상의 제안이 제출, 리뷰, 수용되었습니다.

Swift 2.2의 변경 사항은 여러분의 코드에 직접적인 영향을 미치며, 이 글에서는 무엇이 바뀌었고 왜 바뀌었는지, 코드 예제와 함께 Swift 2.2 새 구문으로 빠르게 전환하는 방법을 안내합니다.

참고로, "지원 중단(deprecation)"은 함수나 언어 기능이 더 이상 사용을 권장하지 않으며 나중에 완전히 제거될 것임을 의미합니다. 실제로 Swift가 현재는 컴파일러 경고를, 향후에는(아마도 Swift 3에서) 컴파일러 오류를 발생시킵니다.

### 컴파일 타임 Swift 버전 검사

Swift의 지속적인 발전은 앱 개발자에게도 따라가기 어려울 수 있지만, 라이브러리 개발자에게는 더 어렵습니다 &mdash; 수천 명이 라이브러리를 사용한다면, 모든 사람이 사용하는 버전에 상관없이 작업물을 쓸 수 있도록 여러 Swift 버전을 지원해야 할 수 있습니다.

Swift 2.2는 크로스 버전 호환성을 간단하게 만드는 새로운 컴파일러 지시문을 도입합니다: 이제 컴파일러가 특정 Swift 언어 버전을 지원하는 경우에만 읽어야 하는 코드 블록을 지정할 수 있습니다. 예를 들어:

~~~swift
#if swift(>=3.0)
print("Running Swift 3.0 or later")
#else
print("Running Swift 2.2 or earlier")
#endif
~~~

이것은 Swift 2에서 도입된 `#available` 구문과 다릅니다. `#available`은 런타임 검사인 반면, 이 새 기능은 컴파일 타임입니다. 따라서 언어 버전 검사에 실패한 코드는 사실상 보이지 않습니다. 원한다면 이런 코드를 작성할 수도 있습니다:

~~~swift
#if swift(>=3.0)
print("Running Swift 3.0 or later")
#else
BRING BACK WASH HE WAS MY FAVORITE
#endif
~~~

Swift 컴파일러가 3.0 이상을 대상으로 하는 한, 대문자 메시지는 컴파일러에 의해 무시되므로 문제없이 컴파일됩니다.

한 가지 주의: Swift 2.1 컴파일러는 `#if swift(>=2.2)`를 이해하지 못하므로 이 기능은 현재 사용할 수 없습니다. 하지만 Swift 3.0이 출시되면, 그리고 이후 모든 버전에서 컴파일 타임 Swift 버전 검사는 유용한 도구가 될 것입니다.

자세한 내용은 이 변경에 대한 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0020-if-swift-version.md)을 참조하세요.

### 컴파일 타임 셀렉터 검사

Swift 2.1에서는 아래와 같은 코드가 문제없이 컴파일되었습니다:

~~~swift
override func viewDidLoad() {
    super.viewDidLoad()

    navigationItem.rightBarButtonItem =
        UIBarButtonItem(barButtonSystemItem: .Add, target: self,
                        action: "addNewFireflyRefernce")
}

func addNewFireflyReference() {
    gratuitousReferences.append("We should start dealing in black-market beagles.")
}
~~~

코드 자체는 문법적으로 올바르지만, 내비게이션 바 버튼이 `addNewFireflyRefernce()` 메서드를 호출하므로 앱이 충돌합니다 &mdash; "reference"의 E가 하나 빠져 있습니다. 이런 단순한 오타가 쉽게 버그를 유발할 수 있으므로, Swift 2.2는 셀렉터에 문자열을 사용하는 것을 지원 중단하고 새로운 구문 `#selector`를 도입합니다.

`#selector`를 사용하면 호출하려는 메서드가 실제로 존재하는지 컴파일 타임에 검사합니다. 더 나아가 메서드가 존재하지 _않으면_ 컴파일 오류가 발생합니다: Xcode가 앱 빌드를 거부하여 또 다른 잠재적 버그 원인을 제거합니다.

`#selector`를 사용하여 이전 코드를 다시 작성하면:

~~~swift
override func viewDidLoad() {
    super.viewDidLoad()

    navigationItem.rightBarButtonItem =
        UIBarButtonItem(barButtonSystemItem: .Add, target: self,
                        action: #selector(addNewFireflyRefernce))
}

func addNewFireflyReference() {
    gratuitousReferences.append("Curse your sudden but inevitable betrayal!")
}
~~~

코드를 빌드하면 컴파일러가 "Use of unresolved identifier 'addNewFireflyRefernce'" 오류를 반환합니다!

자세한 내용은 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0022-objc-selectors.md)이나 이론적 근거를 설명하는 [swift-evolution-announce 게시물](https://lists.swift.org/pipermail/swift-evolution-announce/2016-January/000026.html)을 참조하세요.

### 더 많은 키워드를 인수 레이블로 사용

Swift에는 `class`, `func`, `let`, `public` 등 특별한 의미를 갖고 식별자로 사용할 수 없는 많은 키워드가 있습니다. Swift는 항상 키워드를 인수 레이블로 사용할 수 있었지만, 백틱으로 감싸야 했습니다:

~~~swift
func visitCity(name: String, `in` state: String) {
    print("I'm going to visit \(name) in \(state)")
}

visitCity("Nashville", `in`: "Tennessee")
~~~

Swift 2.2부터 `inout`, `var`, `let`을 제외한 모든 키워드를 인수 레이블로 사용할 수 있습니다. 백틱으로 키워드를 사용한 코드가 있다면, Xcode Fix-it이 이를 제거하라고 제안합니다. 이제 이런 코드가 가능합니다:

~~~swift
func visitCity(name: String, in state: String) {
    print("I'm going to visit \(name) in \(state)")
}

visitCity("Nashville", in: "Tennessee")
~~~

자세한 내용은 이 변경에 대한 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0001-keywords-as-argument-labels.md)을 참조하세요.

### 튜플 비교 내장 지원

튜플은 Swift의 기본 데이터 타입이며, 함수에서 여러 값을 반환할 수 있는 등 많은 이점을 가져옵니다. Swift 2.2는 두 튜플의 동일성을 비교하는 기능을 도입하여, 한 튜플의 각 요소를 다른 튜플의 대응하는 요소와 비교하고 모든 요소가 일치하면 true를 반환합니다.

예를 들어, 아래 코드는 "No match"를 출력합니다:

~~~swift
let singer = (first: "Taylor", last: "Swift")
let alien = (first: "Justin", last: "Bieber")

if singer == alien {
    print("They match! That explains why you never see them together…")
} else {
    print("No match.")
}
~~~

Swift 2.2의 튜플 비교는 arity 6까지 동작합니다. 이는 튜플에 6개 이하의 요소가 포함된 경우에만 비교할 수 있다는 뜻입니다.

한 가지 주의: Swift 2.2는 동일성 검사 시 요소 이름을 무시합니다. 아래 코드에서 `singer`와 `bird`는 동일한 것으로 간주됩니다:

~~~swift
let singer = (first: "Taylor", last: "Swift")
let bird = (name: "Taylor", breed: "Swift")

if singer == bird {
    print("This explains why she sings so well.")
} else {
    print("No match.")
}
~~~

자세한 내용은 이 변경에 대한 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0015-tuple-comparison-operators.md)을 참조하세요.

### 튜플 splat 구문 지원 중단

튜플에 관해 조금 더 이야기하면, Swift 2.2는 이 멋진 이름 때문에만 언급할 가치가 있는 거의 사용되지 않던 기능을 지원 중단합니다. Swift 2.1 이전에는 주의 깊게 만든 튜플로 함수의 매개변수를 채울 수 있었습니다. 함수가 두 개의 매개변수를 받는다면, 올바른 타입과 요소 이름을 가진 2요소 튜플로 호출할 수 있었습니다:

~~~swift
func describePerson(name: String, age: Int) {
    print("\(name) is \(age) years old")
}

let person = ("Malcolm Reynolds", age: 49)
describePerson(person)
~~~

이 구문은 &mdash; 애칭으로 "튜플 splat 구문"이라 불리는 &mdash; Swift의 자기 문서화, 가독성 좋은 스타일과 정반대이므로 Swift 2.2에서 지원 중단되었습니다.

자세한 내용은 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0029-remove-implicit-tuple-splat.md)이나 이론적 근거를 설명하는 [swift-evolution-announce 게시물](https://lists.swift.org/pipermail/swift-evolution-announce/2016-February/000033.html)을 참조하세요.

### C 스타일 <code>for</code> 루프 지원 중단

Swift에 여러 관용적 루프 옵션이 있지만, C 스타일 for 루프도 언어의 일부로 가끔 사용되었습니다:

~~~swift
for var i = 0; i < 10; i++ {
    print(i)
}
~~~

Swift 2.2에서 지원 중단되었으며 Swift 3.0에서 완전히 제거됩니다 &mdash; 세미콜론을 입력하지 않아도 되는 세상에 한 걸음 더 가까워졌습니다.

Xcode를 사용하면 C 스타일 for 루프를 현대적 Swift로 변환하는 Fix-it이 제공될 수 있습니다. 위의 경우, 결과는 범위를 사용합니다:

~~~swift
for i in 0 ..< 10 {
    print(i)
}
~~~

하지만 Fix-it의 기능에는 한계가 있으므로 직접 작업해야 할 부분도 있습니다. 예를 들어, 아래 두 루프는 현재 Fix-it이 도와주지 못합니다:

~~~swift
for var i = 10; i > 0; i-- {
    print(i)
}

for var i = 0; i < 10; i += 2 {
    print(i)
}
~~~

첫 번째 경우 `(1...10).reverse()`를 사용하여 역방향 범위를 만들어야 합니다. 이것은 `i in 10...1`로 작성하는 것과 _같지 않으며_, 이는 컴파일은 되지만 런타임에 충돌합니다. 두 번째 경우 `stride(to:by:)`를 사용하여 2씩 증가해야 합니다. Swift 2.2에서 두 루프를 올바르게 다시 작성하면:

~~~swift
for i in (1...10).reverse() {
    print(i)
}

for i in 0.stride(to: 10, by: 2) {
    print(i)
}
~~~

자세한 내용은 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0007-remove-c-style-for-loops.md)이나 이론적 근거를 설명하는 [swift-evolution-announce 게시물](https://lists.swift.org/pipermail/swift-evolution-announce/2015-December/000001.html)을 참조하세요.

### `++`와 `--` 지원 중단

C 스타일 for 루프를 사용했다면, 이 다음 변경이 더 놀라울 수 있습니다: `++`와 `--`도 전위 및 후위 연산자 모두 지원 중단됩니다. `for var i = 0; i < 10; i++` 같은 코드에 하나가 아닌 _두 개_의 지원 중단이 포함되어 있다는 뜻이며, 빠르게 변화하는 Swift 세계에서도 상당히 주목할 만합니다.

이 변경으로 아래 코드가 모두 지원 중단되었으며 Swift 3에서 완전히 동작을 멈춥니다:

~~~swift
i++
i--
++i
--i
i = i++
~~~

대신 `i += 1` 또는 `i -= 1`을 사용해야 하며, Xcode가 위의 각 예에 대해 Fix-it을 제공합니다. 솔직히 말하면: `i = i++`에 대한 Fix-it은 컴파일러 오류를 발생시키는데, 이는 크게 놀랍지 않습니다 &mdash; `i = i++`가 무엇을 해야 하는지 자체가 모호하니까요.

이 변경에 단일한 이유는 없으며, 여러 작은 이유가 합쳐진 것입니다:

* `i++`는 `i += 1`보다 약간 짧을 뿐입니다.
* Swift 입문자가 `++`와 `--`에 익숙하지 않다면 학습 곡선만 가파르게 만듭니다.
* `++`와 `--`가 흔히 사용되는 C 스타일 루프도 지원 중단됩니다.
* 이 연산자의 결과를 사용하는 것은 전위/후위 여부에 따라 달라져 혼동을 줍니다.

이 변경에 대한 Swift Evolution 제안에 이론적 근거를 간결하게 요약하는 인용이 있습니다: "이것들이 없다면 Swift 3에 추가할 것인가?"라는 질문에 실패한다는 것입니다. \[[1](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0004-remove-pre-post-inc-decrement.md)\]

자세한 내용은 이 변경에 대한 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0004-remove-pre-post-inc-decrement.md)을 참조하세요.

### `var` 매개변수 지원 중단

Swift 2.2 이전에는 함수 내에서 매개변수를 수정하려면 `var`로 선언할 수 있었습니다:

~~~swift
func greet(var name: String) {
    name = name.uppercaseString
    print("Hello, \(name)!")
}

var name = "Taylor"
greet(name)
print("After function, name is \(name)")
~~~

유용한 단축키이긴 했지만 혼동을 주었습니다: 마지막 `print()` 문이 "Taylor"를 출력할까요, "TAYLOR"를 출력할까요? `inout` 키워드의 존재로 더 혼란스러워집니다: 이 예에서 `var` 대신 `inout`을 사용하고 앰퍼샌드 하나를 추가하면:

~~~swift
func greet(inout name: String) {
    name = name.uppercaseString
    print("Hello, \(name)!")
}

var name = "Taylor"
greet(&name)
print("After function, name is \(name)")
~~~

`var` 예제와 `inout` 예제는 다른 출력을 생성합니다. `var` 매개변수의 변경은 함수 내에서만 적용되는 반면 `inout` 매개변수의 변경은 원래 값에 직접 영향을 미치기 때문입니다.

Swift 2.2에서는 함수 매개변수의 `var` 키워드를 지원 중단하여 이 혼란을 해소하며, Swift 3.0에서 완전히 제거됩니다. 이전 동작을 복제하려면 함수 내에서 직접 복사본을 만들면 됩니다:

~~~swift
func greet(name: String) {
    let uppercaseName = name.uppercaseString
    print("Hello, \(uppercaseName)!")
}
~~~

자세한 내용은 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0003-remove-var-parameters.md)이나 이론적 근거를 설명하는 [swift-evolution-announce 게시물](https://lists.swift.org/pipermail/swift-evolution-announce/2016-January/000027.html)을 참조하세요.

### 디버그 식별자 이름 변경

Swift 컴파일러는 디버깅에 유용한 몇 가지 심볼을 자동으로 제공합니다. 이전에는 "스크리밍 스네이크 케이스"였으므로, `__FILE__`은 현재 Swift 파일 이름으로, `__LINE__`은 줄 번호로 대체되었습니다. Swift 2.2에서 이전 식별자들은 지원 중단되고 `#file`, `#line`, `#column`, `#function`으로 교체되어 "# 은 여기서 컴파일러 치환 로직을 호출한다"는 규칙을 도입합니다. \[[2](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0028-modernizing-debug-identifiers.md)\]

이전과 새 구문을 보여주는 예시입니다:

~~~swift
func visitCity(name: String, in state: String) {
    // old - deprecated!
    print("This is on line \(__LINE__) of \(__FUNCTION__)")

    print("I'm going to visit \(name) in \(state)")

    // new - shiny!
    print("This is on line \(#line) of \(#function)")
}
~~~

다른 많은 변경과 마찬가지로 Xcode에 코드를 올바르게 업데이트하는 Fix-it이 있습니다.

자세한 내용은 [Swift Evolution 제안](https://github.com/swiftlang/swift-evolution/blob/master/proposals/0028-modernizing-debug-identifiers.md)이나 이론적 근거를 설명하는 [swift-evolution-announce 게시물](https://lists.swift.org/pipermail/swift-evolution-announce/2016-February/000030.html)을 참조하세요.

이 글에서는 대부분의 개발자에게 영향을 미칠 변경 사항을 다루었지만, 개선된 컴파일러 메시지 및 성능 향상과 함께 다른 작은 변경도 도입되었습니다. 전체 변경 사항과 토론 링크, Linux 설치 방법이 포함된 [공식 릴리스 노트](/blog/swift-2-2-released/)를 확인하세요.
